# コメント

## 本番コード

`app/src/main` にコメントを書かない。行コメント、ブロックコメント、KDoc のいずれも書かない。

例外はツールや法的要件が要求するものだけ（`@Suppress` の理由、ライセンス表記など）。

説明を書きたくなったら、それはコードで表現できていないという合図として扱う。
名前、値オブジェクト、早期リターン、関数の分割で意図を表す。
背景・トレードオフ・採用しなかった案は PR の説明か `docs/` へ書き、コードには残さない。

## テストコード

`app/src/test` と `app/src/androidTest` に書いてよいコメントは次の 2 種類だけ。

1. AAA ラベル: `// Arrange` / `// Act` / `// Assert` / `// Act & Assert`
2. 検証観点: `// Act` または `// Act & Assert` の直下に 1 行

検証観点は **英語** で書く。

```kotlin
// Act
// Boundary: blank input is rejected
```

定数の根拠、fixture の説明、ヘルパー関数の KDoc は書かない。
値の意図はテスト関数名と検証観点で表す。

観点の種類と命名規則は [`docs/unit-test.md`](../../docs/unit-test.md) を正本とする。

## 実装前の確認

新規ファイルを追加する前に、同じ package の既存ファイルを開いて記述の粒度を確認する。
